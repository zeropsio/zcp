# Phase 6 Post-Work Audit: RemoteURL Cache Refresh

## 1. Verdict

SHIP-WITH-NOTES. Phase 6 implements the intended live-remote-first behavior: export reads the live git remote before bundle composition, uses that value in `ops.BundleInputs.RepoURL`, refreshes `ServiceMeta.RemoteURL`, and surfaces stale-cache drift as a warning rather than a blocker (`internal/tools/workflow_export.go:120`, `internal/tools/workflow_export.go:133`, `internal/tools/workflow_export.go:155`, `internal/tools/workflow_export.go:163`, `internal/tools/workflow_export.go:173`). ## 5. Race/concurrency

Flag as low-to-medium residual risk. `WriteServiceMeta` creates the services directory, marshals JSON, writes to a fixed temp path of `<hostname>.json.tmp`, then renames it over the target path (`internal/workflow/service_meta.go:236`, `internal/workflow/service_meta.go:241`, `internal/workflow/service_meta.go:246`, `internal/workflow/service_meta.go:247`, `internal/workflow/service_meta.go:248`, `internal/workflow/service_meta.go:251`). That gives atomic replacement at the final path, but there is no visible lock, compare-and-swap, or unique temp file in `WriteServiceMeta` itself (`internal/workflow/service_meta.go:235`, `internal/workflow/service_meta.go:247`, `internal/workflow/service_meta.go:248`, `internal/workflow/service_meta.go:251`). Concurrent export calls with the same live URL are likely benign, but concurrent writes to different fields on the same meta can still lose updates because the helper mutates and rewrites a previously read full meta (`internal/tools/workflow_export.go:91`, `internal/tools/workflow_export_probe.go:244`, `internal/tools/workflow_export_probe.go:245`).
