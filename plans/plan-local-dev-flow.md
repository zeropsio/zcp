# Plan: Local Development Flow for ZCP

**Date**: 2026-03-26
**Status**: Draft v3 — deep audit corrected (review-2: 2 critical fixes)
**Based on**: `plans/analysis-local-dev-flow.analysis-1.md` (4-agent deep analysis), gap validation (3 agents), `plans/plan-local-dev-flow.review-1.md` (4-agent deep audit), `plans/plan-local-dev-flow.review-2.md` (4-agent deep audit: strategy key fix, transition message fix)
**Verified facts**: VPN hostname resolution (live test macOS), env var unavailability via VPN, zcli push from local, ${hostname_varName} container-level resolution, wg-quick auto-sudo mechanism, zcli cli.data structure, pollDeployBuild API-based (env-agnostic), zcli --working-dir relative paths, zcli positional arg accepts hostname or ID, --service-id kebab-case

## 1. Goal

Enable ZCP to orchestrate Zerops projects from the user's local machine. Code lives locally, dev server runs locally, managed services (DB, cache) run on Zerops (accessible via VPN). No SSHFS mounts, no SSH-based deploys.

**One sentence**: User develops locally with hot reload, ZCP provides env vars + health verification + one-command deploy to Zerops.

---

## 2. Design Decisions (confirmed)

| # | Decision | Rationale |
|---|----------|-----------|
| D1 | No dev service on Zerops in local mode | User's machine IS dev. Zerops dev service running `zsc noop` is wasteful. |
| D2 | Project-scoped integration token required | Zerops tokens have three scope levels (full, read, custom per-project). ZCP requires "Custom access per project" with exactly one project → `ListProjects` returns exactly 1. Account-wide tokens with access to multiple projects are rejected with guidance. |
| D3 | No `ZCP_PROJECT_ID` env var | Not needed — project-scoped token handles scoping. Keep `.mcp.json` minimal. |
| D4 | Deploy via conditional registration — same tool name `zerops_deploy`, different schema per environment | Agent always calls `zerops_deploy`. Container mode registers with SSH schema (sourceService, workingDir=/var/www). Local mode registers with local schema (workingDir=CWD, no sourceService). No phantom parameters. |
| D5 | ZCP does NOT manage dev server process | Too many variants (Valet, IDE, Docker, CLI). ZCP provides data + verification only. |
| D6 | Universal local config: port + envFile | Two fields. Works for all runtimes. |
| D7 | .env file generated from `zerops_discover` | Most universal env var format. How user loads it = their concern (guidance helps). |
| D8 | VPN: guidance + diagnostics, optional auto-connect via sudoers | ZCP cannot start VPN without admin privileges. Opt-in sudoers setup enables auto-connect. |
| D9 | Both `hostname` and `hostname.zerops` work over VPN | Live-verified 2026-03-26 on macOS. DNS search domain handles it. |
| D10 | Connection diagnostics via existing tools, no VPN check in code | Agent uses `nc`, `zerops_discover`, `/status` endpoint to diagnose. No TCP probe in ZCP. |
| D11 | Plan format unchanged — engine adapts per environment | Agent submits identical plan in container and local mode. Engine decides what to create. |
| D12 | Deploy strategies unchanged — push-dev = push from local to target | push-dev, ci-cd, manual work in both environments. Only the deploy mechanism changes. |
| D13 | Local health check = agent curl via Bash, not zerops_verify | zerops_verify is for Zerops services. Local dev server checked by agent directly. |
| D14 | Build logs via API polling, same as SSH mode | pollDeployBuild() is env-agnostic. deployLocal() returns BUILD_TRIGGERED, API does the rest. |
| D15 | ServiceMeta carries `environment` field | Meta self-describes whether it was created in container or local mode. Consumers (orientation, checkers) branch on this field instead of inferring from context. |

## 3. Architecture: What Changes

### 3.1 Current flow (container mode)

```
zcpx container (Zerops)
  ├── .zcp/state/          ← sessions, ServiceMeta
  ├── /var/www/{host}/     ← SSHFS mounts to dev services
  └── ZCP process
       ├── SSHDeployer     ← SSH into container, run zcli push
       ├── Mounter         ← SSHFS mount/unmount
       └── zerops_deploy   ← registered via RegisterDeploySSH
```

### 3.2 New flow (local mode)

```
User's machine
  ├── ~/projects/myapp/
  │   ├── .zcp/state/      ← sessions, ServiceMeta, local.json
  │   ├── .mcp.json        ← ZCP_API_KEY (project-scoped token)
  │   ├── .env             ← generated by ZCP from zerops_discover
  │   ├── zerops.yml       ← for deploy target on Zerops
  │   └── src/             ← user's code
  └── ZCP process
       ├── sshDeployer=nil  ← no SSH
       ├── mounter=nil      ← no SSHFS
       └── zerops_deploy    ← registered via RegisterDeployLocal
```

### 3.3 Key architectural change — conditional registration

The guard in `server.go:108-110` becomes a **router**, not a gate:

**Current**:
```go
if s.sshDeployer != nil {
    tools.RegisterDeploy(...)  // Only in container mode — local mode has NO deploy tool
}
```

**New**:
```go
if s.sshDeployer != nil {
    tools.RegisterDeploySSH(s.server, s.client, projectID, s.sshDeployer, s.authInfo, s.logFetcher, s.rtInfo)
} else {
    tools.RegisterDeployLocal(s.server, s.client, projectID, s.authInfo, s.logFetcher)
}
```

Both register the same tool name `zerops_deploy` but with different:
- **Input struct** (schema) — SSH has sourceService; local does not
- **Description** — SSH mentions SSH; local mentions zcli push
- **Handler** — SSH calls `ops.DeploySSH()`; local calls `ops.DeployLocal()`

`pollDeployBuild()` is shared — extracted to `tools/deploy_poll.go`.

Why NOT internal routing in one function:
- `sourceService` is meaningless in local mode — showing it in schema is a lie
- `workingDir` has different default and semantics per mode
- One tool with phantom parameters → LLM confusion at runtime

### 3.4 Plan format: unchanged, engine adapts

Agent submits **identical plan** regardless of environment:

```json
{"runtime": {"devHostname": "appdev", "type": "nodejs@22", "bootstrapMode": "standard"},
 "dependencies": [{"hostname": "db", "type": "postgresql@16", "resolution": "CREATE"}]}
```

Engine behavior per environment:

| Step | Container | Local |
|------|-----------|-------|
| Provision | Create appdev + appstage + db | Create **only appstage + db** (skip appdev) |
| Mount | SSHFS mount appdev | **Skip** |
| Generate | Write to `/var/www/appdev/` | Write to **local CWD** |
| Deploy | SSH: appdev → appstage | zcli push: **local → appstage** |
| Close | ServiceMeta: hostname=appdev, stage=appstage, env=container | ServiceMeta: **hostname=appstage**, env=local |

Validation (`validate.go:180-190`) passes identically — `devHostname` ends in "dev", `StageHostname()` derives "appstage".

### 3.5 ServiceMeta per environment

New `environment` field added to ServiceMeta:

```go
type ServiceMeta struct {
    Hostname         string `json:"hostname"`
    Mode             string `json:"mode,omitempty"`
    StageHostname    string `json:"stageHostname,omitempty"`
    DeployStrategy   string `json:"deployStrategy,omitempty"`
    Environment      string `json:"environment,omitempty"`   // "container" or "local"
    BootstrapSession string `json:"bootstrapSession"`
    BootstrappedAt   string `json:"bootstrappedAt"`
}
```

Container:
```json
{"hostname": "appdev", "mode": "standard", "stageHostname": "appstage", "environment": "container", "deployStrategy": "push-dev"}
```

Local:
```json
{"hostname": "appstage", "mode": "standard", "stageHostname": "", "environment": "local", "deployStrategy": "push-dev"}
```

Why `hostname=appstage` in local mode:
- `filterStaleMetas` (router.go:76) checks `live[m.Hostname]` — appstage exists on Zerops ✓
- If hostname were "appdev", it would be filtered out (appdev doesn't exist) → entire post-bootstrap system breaks
- Semantically correct: appstage IS the deploy target, user's laptop IS dev

`BuildDeployTargets()`: mode=standard, stageHostname="" → `deployRoleFromMode` returns "simple" (deploy.go:170-174). Target = `[{appstage, simple, push-dev}]`. Deploy goes directly to appstage.

Why `environment` field:
- `writeServiceBlock` (instructions_orientation.go:112) outputs mount paths + SSH commands — wrong for local
- `workflow_checks_deploy.go:173` checks zerops.yml on SSHFS mount path — wrong for local
- Meta self-describes its context → consumers branch cleanly, no fragile inference

### 3.6 writeBootstrapOutputs — environment branch

`bootstrap_outputs.go:23` currently writes `hostname := target.Runtime.DevHostname` unconditionally.

**Critical**: Strategy lookup at line 27 (`state.Bootstrap.Strategies[hostname]`) must use `DevHostname` as key — the Strategies map is populated by the agent using DevHostname from plan targets. The meta hostname must be separated from the strategy lookup key.

```go
func (e *Engine) writeBootstrapOutputs(state *WorkflowState) {
    for _, target := range plan.Targets {
        devHostname := target.Runtime.DevHostname
        stageHostname := target.Runtime.StageHostname()

        // Strategy lookup ALWAYS by DevHostname — agent stores strategies under DevHostname
        strategy := state.Bootstrap.Strategies[devHostname]

        // Meta hostname: local mode writes appstage (dev doesn't exist), container writes appdev
        metaHostname := devHostname
        if e.environment == EnvLocal && stageHostname != "" {
            metaHostname = stageHostname
            stageHostname = ""
        }

        meta := &ServiceMeta{
            Hostname:      metaHostname,
            StageHostname: stageHostname,
            Mode:          target.Runtime.EffectiveMode(),
            DeployStrategy: strategy,
            Environment:   string(e.environment),
            // ...
        }
    }
}
```

Same hostname separation in `writeProvisionMetas()` (no strategy lookup there, but consistent pattern).

### 3.7 BuildTransitionMessage — environment-aware display

`bootstrap_guide_assembly.go:79-103` uses `t.Runtime.DevHostname` for display and strategy lookup. In local mode, DevHostname ("appdev") doesn't exist on Zerops — confusing for the user.

Add `env Environment` parameter to `BuildTransitionMessage`:

```go
func BuildTransitionMessage(state *WorkflowState, env Environment) string {
    for _, t := range targets {
        displayName := t.Runtime.DevHostname
        if env == EnvLocal {
            displayName = t.Runtime.StageHostname()  // "appstage" — the real target
        }
        // Use displayName for user-facing text
        // Use DevHostname for Strategies map lookup (strategies keyed by DevHostname)
    }
}
```

Caller in `engine.go` already has `e.environment` — pass it through.

## 5. State & Evidence

### 5.1 Directory structure

```
~/projects/myapp/
  ├── .mcp.json                        ← Claude Code MCP config (ZCP_API_KEY)
  ├── .zcp/
  │   └── state/
  │       ├── sessions/                ← ephemeral workflow sessions
  │       ├── services/appstage.json   ← ServiceMeta (persistent)
  │       ├── registry.json            ← session registry
  │       └── local.json               ← local dev config (NEW)
  ├── .env                             ← generated env vars (NEW)
  ├── .gitignore                       ← includes .env
  ├── zerops.yml                       ← build/deploy config
  └── (user's code)
```

### 5.2 `.mcp.json` — auth binding

```json
{
  "mcpServers": {
    "zcp": {
      "command": "zcp",
      "env": {
        "ZCP_API_KEY": "<project-scoped-integration-token>"
      }
    }
  }
}
```

Token created as "Custom access per project" in Zerops GUI → Full access to exactly one project. Zerops supports three token scope levels (full access, read access, custom per-project) — ZCP requires the custom per-project scope.

### 5.3 `.zcp/state/local.json` — local dev config (NEW)

```json
{
  "port": 3000,
  "envFile": ".env"
}
```

Two fields. Created during bootstrap generate step. Port from zerops.yml `run.ports`. Used by guidance for localhost health check hint.

### 5.4 Multi-project

Each project directory has its own `.mcp.json` + `.zcp/state/`. No collision:
```
~/projects/app1/.mcp.json  → token A → project A
~/projects/app2/.mcp.json  → token B → project B
```

VPN is one per machine — user switches manually (`zcli vpn up <projectId>`).

### 5.5 Monorepo

`.zcp/state/` always at CWD where Claude is opened (`server.go:89-90`). `workingDir` in `zerops_deploy` is the zcli push source path, separate from state location.

```
~/monorepo/
  ├── .zcp/state/services/frontstage.json
  ├── .zcp/state/services/apistage.json
  ├── frontend/zerops.yml  (setup: frontstage)
  ├── backend/zerops.yml   (setup: apistage)
```

Deploy: `zerops_deploy targetService="frontstage" workingDir="./frontend"`

Note: zcli push has a `--setup` flag that selects which zerops.yml entry to deploy when the file contains multiple `setup:` entries. For monorepo with separate zerops.yml per subfolder, `workingDir` alone is sufficient. For monorepo with a single root zerops.yml containing multiple entries, the `--setup` flag would be needed (future enhancement — v1 uses separate files per subfolder).

## 6. Bootstrap Flow (local mode)

### Step 1: Discover (no change)

Identical to container mode. API-based, environment-agnostic. Agent submits same plan format.

### Step 2: Provision (environment branch)

- **Import**: Environment-aware — skip dev service creation in local mode
  - Container standard: import appdev + appstage + managed
  - Local standard: import **only appstage + managed**
  - Simple/dev-only: same in both environments
- **Mount**: SKIP in local mode (mounter=nil, guidance doesn't suggest it)
- **Env var discovery**: Same — `zerops_discover includeEnvs=true`

### Step 3: Generate (guidance change)

Agent writes files to **local CWD** instead of SSHFS mount path.

- `zerops.yml` — for deploy target (appstage: real start, healthCheck)
- App code — if new project
- **`.env` file** (NEW) — resolved env var values from discover
- **`.gitignore`** — includes `.env` (if not present)

**`.env` generation**:
```
# Generated by ZCP from zerops_discover
# VPN required: zcli vpn up <projectId>
# WARNING: Contains secrets. Do not commit.

# db (postgresql@16)
db_host=db
db_port=5432
db_user=db
db_password=<actual-password>
db_connectionString=postgresql://db:<actual>@db:5432/db

# cache (valkey@7.2)
cache_host=cache
cache_port=6379

# storage (object-storage)
storage_apiUrl=https://storage.app-prg1.zerops.io
storage_accessKeyId=<actual>
storage_secretAccessKey=<actual>
storage_bucketName=<actual>
```

Hostnames use plain service hostname (works over VPN — verified). S3 object storage uses HTTPS apiUrl — **UNVERIFIED whether accessible without VPN** (docs say "accessible remotely over internet" but TLS handshake to generic endpoint `storage.app-prg1.zerops.io:443` failed; per-service apiUrl may differ — needs E2E verification with real object-storage service before release).

### Step 4: Deploy (major change)

1. **VPN guidance**: "Run in your terminal: `zcli vpn up <projectId>`" (project ID from `authInfo.ProjectID`)
   - If `CanAutoVPN()` → agent runs it via Bash automatically
2. **Dev server guidance**: Runtime-aware hint from knowledge
3. **Local health check**: Agent runs `curl localhost:{port}/health` via Bash
4. **Deploy to target**: `zerops_deploy targetService="appstage"` → `deployLocal()` → `zcli push`
5. **Verify target**: `zerops_verify serviceHostname="appstage"` — API + subdomain HTTP (same as container mode)

#### Dev server guidance (universal, not code)

Agent asks user how they run their dev server. ZCP does NOT start it. Runtime-aware hints injected via knowledge system:

| Runtime | Typical hint |
|---------|-------------|
| PHP + Valet/Herd | "Already running — navigate to project URL" |
| PHP without | "Run: `php artisan serve`" |
| Node.js | "Run: `npm run dev`" |
| Go | "Run: `go run .` (or `air` for hot reload)" |
| Python/Django | "Run: `python manage.py runserver`" |
| Java | "Run from IDE or: `mvn spring-boot:run`" |
| Generic | "Start your dev server, I'll verify health" |

#### Connection error diagnostics

When `/status` shows connection error to managed service, agent diagnoses:

```
1. zerops_discover service="db" → is service RUNNING on Zerops?
   - STOPPED → zerops_manage action="start"
   - ACTIVE → continue
2. bash: nc -zv db 3306 -w 3 → is VPN working?
   - timeout → "VPN not connected. Run: zcli vpn up <projectId>"
   - connected → VPN ok, check credentials
3. Compare .env vs zerops_discover includeEnvs=true → credentials match?
   - Mismatch → regenerate .env
   - Match → app config issue (wrong env var name, wrong driver, etc.)
```

### Step 5: Close (environment branch)

- **Container**: ServiceMeta `{hostname: "appdev", stageHostname: "appstage", environment: "container"}`
- **Local**: ServiceMeta `{hostname: "appstage", stageHostname: "", environment: "local"}`

### 7.2 Local tool registration

```go
// tools/deploy_local.go

type DeployLocalInput struct {
    TargetService string `json:"targetService"        jsonschema:"Hostname of the service to deploy to."`
    WorkingDir    string `json:"workingDir,omitempty"  jsonschema:"Local path to push from. Default: current directory."`
    IncludeGit    bool   `json:"includeGit,omitempty"  jsonschema:"Include .git directory in the push (-g flag)."`
}

func RegisterDeployLocal(srv, client, projectID, authInfo, logFetcher) {
    mcp.AddTool(srv, &mcp.Tool{
        Name:        "zerops_deploy",  // SAME tool name as SSH variant
        Description: "Push local code to Zerops — blocks until build completes. " +
            "Requires zerops.yml and zcli installed. " +
            "Set targetService to the Zerops service hostname.",
        Annotations: &mcp.ToolAnnotations{
            Title:           "Deploy code to a service",
            DestructiveHint: boolPtr(true),
        },
    }, func(ctx, req, input DeployLocalInput) {
        result, err := ops.DeployLocal(ctx, client, projectID, *authInfo,
            input.TargetService, input.WorkingDir, input.IncludeGit)
        if err != nil { return convertError(err) }

        onProgress := buildProgressCallback(ctx, req)
        pollDeployBuild(ctx, client, projectID, result, onProgress, logFetcher, nil)

        return jsonResult(result)
    })
}
```

### 7.3 New function: `ops.DeployLocal()`

Location: `internal/ops/deploy_local.go` (new file)

```go
func DeployLocal(
    ctx context.Context,
    client platform.Client,
    projectID string,
    authInfo auth.Info,
    targetService string,
    workingDir string,
    includeGit bool,
) (*DeployResult, error) {
    // 1. Validate zcli in PATH (exec.LookPath)
    // 2. Resolve targetService hostname → service ID via API (shared resolveServiceID)
    // 3. Default workingDir to "." if empty
    // 4. Validate zerops.yml exists at workingDir (os.Stat)
    // 5. Run ValidateZeropsYml(workingDir, targetService) — local path, NOT mount path
    // 6. exec.Command: zcli login <token>
    // 7. exec.Command: zcli push <targetService> --working-dir <path> [-g]
    //    (positional arg = hostname, simpler than --service-id flag)
    // 8. If exit 0: return DeployResult{Status: BUILD_TRIGGERED, Mode: "local", TargetServiceID: id}
    // 9. If exit != 0: classify error (auth failed, push failed, zcli error)
}
```

Key differences from SSH path:
- Uses `zcli push <hostname>` (positional arg, verified) instead of `--serviceId` (camelCase was wrong — actual flag is `--service-id` kebab-case, but positional arg is simpler)
- Calls `ValidateZeropsYml(workingDir, targetService)` with actual local path (not SSHFS mount)
- No git init/commit dance (zcli push from local handles git itself)
- No sourceService concept (code lives locally, not on another service)
- Returns `Mode: "local"` in DeployResult

### 7.9 Error classification

| Error | Detection | Message |
|-------|-----------|---------|
| zcli not in PATH | `exec.LookPath` fails | "Install zcli" |
| zcli auth failed | exit != 0 + stderr contains "login" | "zcli login failed — check token" |

| Build pipeline failed | API polling: status != ACTIVE | buildLogs from API (same as SSH) |
| zerops.yml missing | `os.Stat` fails | "zerops.yml not found at {path}" |

## 8. Env Var Bridge

### 8.1 Problem

VPN provides network access but NOT env vars (verified). Local app needs actual values.

### 8.2 Solution: `.env` generation

New function in `internal/ops/env_export.go`:

```go
func FormatEnvFile(services []ServiceEnvGroup, projectID string) string
```

### 8.3 zerops.yml envVariables vs .env

| File | Purpose | Where | References |
|------|---------|-------|-----------|
| `zerops.yml` envVariables | Zerops container | Zerops | `${db_connectionString}` — resolved by platform |
| `.env` | Local dev | User's machine | Actual values: `postgresql://db:pass@db:5432/db` |

Both coexist. `${hostname_varName}` in zerops.yml works regardless of push source (verified).

### 8.4 S3 object storage

S3 uses HTTPS `apiUrl` endpoint. Zerops docs state object storage is "accessible remotely over internet", but **this has NOT been verified from a local machine without VPN**. TLS handshake to the generic endpoint `storage.app-prg1.zerops.io:443` failed. Per-service apiUrl may use a different, working endpoint. **Needs E2E verification with a real object-storage service before claiming "works without VPN".**

Env vars (`storage_apiUrl`, `storage_accessKeyId`, etc.) included in `.env` automatically from discover regardless.

```

```

### 9.2 Opt-in: `zcp init vpn`

User runs once with admin access. Enables agent to auto-connect VPN.

```bash
$ zcp init vpn
→ Detecting wg-quick: /opt/homebrew/bin/wg-quick
→ Writing sudoers rule (requires sudo)...
→ Verified: passwordless wg-quick works
→ Done.
```

**Windows**: WireGuard tunnel service (runs as SYSTEM). Needs investigation — noted as future work.

### 9.3 Auto-connect detection

```go
canAutoVPN := exec.Command("sudo", "-n", "wg-quick", "--help").Run() == nil
```

If true → agent runs `zcli vpn up <projectId>` via Bash automatically.

## 11. Health Verification
