## 1. Anatomie push-dev dev-to-dev self-deploymentu v kontejnerech

### 1.1 Mise-en-scène

Uživatel má ZCP container v projektu eval-zcp. Každá runtime service v projektu má přimountovaný SSHFS filesystem `/var/www/{hostname}/` → namířený na target service `/var/www/`. Takže ZCP **vidí** a **edituje** kód target service přes tenhle mount. To je ZÁKLAD celého dev-to-dev flow.

```
ZCP container                       uploaddev container (target)
┌──────────────────────┐            ┌─────────────────────────┐
│                      │            │                         │
│  /var/www/uploaddev/ │═══SSHFS═══▶│      /var/www/          │
│    zerops.yaml       │            │        zerops.yaml      │
│    Program.cs        │            │        Program.cs       │
│    App.csproj        │            │        App.csproj       │
│    .git/             │            │        .git/            │
│    ...               │            │        ...              │
└──────────────────────┘            └─────────────────────────┘
           │
           │ User runs zerops_deploy targetService=uploaddev
           ▼
```

SSHFS propojení vytváří iluzi jednoho filesystému. **Ale** je to stále dva různé containery s různými UID namespaces. Proto git-init z mount strany vytvoří root-owned `.git/objects/` (GLC invariant fix). V téhle iluzi deploy funguje zhruba takhle:

### 1.2 Timeline jednoho `zerops_deploy targetService=uploaddev` (push-dev, self-deploy)

**Fáze A — Harness na ZCP** (`internal/tools/deploy_ssh.go:84` handler):

5. `ops.DeploySSH(ctx, ..., sourceService="", targetService="uploaddev", setup="dev", workingDir="")` — sourceService se auto-infers na targetService pro self-deploy (→ `includeGit=true`)

8. `workingDir = "/var/www"` — DEFAULT. Tohle je cesta **uvnitř uploaddev containeru**, kde bude `zcli push` běžet.

15. Git safety-net — `.git/` už existuje (`InitServiceGit` proběhl v bootstrap), takže test short-circuits
16. `git add -A && git commit` — commit v uploaddev containeru (git user.email/name nastaveno během InitServiceGit)
17. `zcli push --service-id <uploaddev> --setup dev -g`:
    - Zabalí `/var/www/` do tar archivu (respektuje `.deployignore`)
    - Uploaduje archiv do Zerops builder infrastruktury (nikoli přímo do runtime!)
    - Blokuje do konce build+deploy pipeline

**Fáze D — Zerops builder** (mimo ZCP, mimo uploaddev — **separátní build container**):

18. Platform provisionuje nový build container s base `dotnet@9`
19. Copy archive → build container's `/var/www/`
20. `prepareCommands` (optional, žádné v tomto case)
21. `buildCommands: [dotnet restore App.csproj, dotnet restore Migrate.csproj]` — dev setup, jen restore, nebuild
22. `deployFiles: ./` → builder archivuje **celý** `/var/www/` ze svého build containeru
23. Archive → interní Zerops storage
24. Build container se smaže

**Fáze E — Runtime deploy** (uploaddev container — v tomto self-deploy je starý container recyklován):

25. Nový runtime container se vytvoří (dotnet@9 base)
26. Download artifact z internal storage
27. `initCommands: [zsc execOnce ... migrate]` — migrace
28. `start: zsc noop --silent` — container idluje
29. Readiness check (none pro dev)
30. Container becomes ACTIVE
31. SSH session, kterou uživatel měl na starém containeru, **zmizí** — `result.Message` zmiňuje "prior SSH sessions are gone"

**Fáze F — Harness poll a návrat**:

32. `pollDeployBuild` (`deploy_poll.go`) poll-loop-uje event status → ACTIVE
33. `FetchBuildWarnings(tag="zbuilder@<event.id>")` — zkusí nalovit warnings z build containeru (ten má svou vlastní log service-stack); scoped by tag identity to this build only
34. `WaitSSHReady` — čeká 30s, až target má SSH
35. Response JSON obsahuje:
    ```json
    {
      "status": "DEPLOYED",
      "message": "Successfully deployed to uploaddev. Run zerops_verify for runtime state.",
      "warnings": [...PRE-FLIGHT WARNINGS from step 9...],    ← F3 false positive persists here
      "buildLogs": [...BUILDER WARNINGS from step 33...],
      "sshReady": true
    }
    ```

### 1.3 Cross-deploy dev → stage je stejný flow s dvěma rozdíly

Pokud uživatel volá `zerops_deploy sourceService=uploaddev targetService=uploadstage setup=prod`:
- `includeGit = false` — nepřenáší `.git/` z dev do stage
- `buildSSHCommand` cílí na uploadstage's service ID
- Ale `sshDeployer.ExecSSH("uploaddev", cmd)` stále běží **na uploaddev containeru**, protože push se provádí z něj
- Build container dostane uploaddev's source code, buildne s `setup: prod` blockem (dotnet publish -o out), deploy artifact do uploadstage
- `mountPath` pro pre-flight je `/var/www/uploaddev` (source, protože my píšeme kód tam)

### 1.4 Klíčový insight: WHERE code lives vs WHERE build happens vs WHERE it runs

| Entity | Cesta | Existuje `./out`? |
|---|---|---|
| Source tree (SSHFS mount, ZCP) | `/var/www/uploaddev/` | NE — user nikdy neběží dotnet publish na ZCP |
| Source tree (uploaddev container) | `/var/www/` | NE — same mirror, jen nedostupný přes SSHFS |
| **Build container (separátní, ephemeral)** | **`/var/www/` uvnitř build containeru** | **ANO** — po `dotnet publish -o out` |
| Runtime (uploaddev container po deployi) | `/var/www/out/` (prod) nebo `/var/www/` (dev with `./`) | ANO (post-deploy) |

**deployFiles jsou definovány nad BUILD CONTAINER filesystem** (platform docs confirm: "files or folders **produced by your build**"). Ne nad source tree. Nikdy ne nad runtime.

## 2. Anatomie dotnet-hello-world recipe

Plný obsah: `internal/knowledge/recipes/dotnet-hello-world.md` (4580 B, gitignored, `zcp sync pull`).

### 2.1 Struktura — dva setupy (prod + dev)

Recipe implementuje **standard mode** pattern: jeden runtime service má dva setupy v jednom zerops.yaml a dva odpovídající hostname v import yaml (např. `appdev` + `appstage`).

### 2.2 PROD setup — multi-app layout s migrací

```yaml
- setup: prod
  build:
    base: dotnet@9
    buildCommands:
      - dotnet publish App/App.csproj -c Release -o out/app         # výstup: out/app/App.dll, out/app/wwwroot/, …
      - dotnet publish Migrate/Migrate.csproj -c Release -o out/migrate   # výstup: out/migrate/Migrate.dll, …
    deployFiles:
      - ./out            # PRESERVES directory; runtime dostane /var/www/out/{app,migrate}/
    cache: true
  deploy:
    readinessCheck:
      httpGet: {port: 8080, path: /}
  run:
    base: dotnet@9
    initCommands:
      - zsc execOnce ${appVersionId} --retryUntilSuccessful -- dotnet ./out/migrate/Migrate.dll
    ports: [{port: 8080, httpSupport: true}]
    envVariables:
      ASPNETCORE_ENVIRONMENT: Production
      ASPNETCORE_URLS: http://0.0.0.0:8080
      DB_NAME: db
      DB_HOST: ${db_hostname}
      DB_PORT: ${db_port}
      DB_USER: ${db_user}
      DB_PASS: ${db_password}
    start: dotnet ./out/app/App.dll
```

**Co tento layout implikuje:**

- Runtime filesystem po deployi: `/var/www/out/app/App.dll` + `/var/www/out/migrate/Migrate.dll` + `/var/www/zerops.yaml` (ne, nedeployovaný — jen `./out`).
- `initCommands` běží **PŘED** startem na každém containeru. `zsc execOnce ${appVersionId}` zajistí, že běží pouze jednou napříč všemi containery (lock na `appVersionId`).
- `start: dotnet ./out/app/App.dll` — spouští App.dll z CWD `/var/www/`.

### 2.3 DEV setup — idle container + SSH iteration

```yaml
- setup: dev
  build:
    base: dotnet@9
    buildCommands:
      - dotnet restore App/App.csproj         # jen restore, cache-uje NuGet packages do build imagu
      - dotnet restore Migrate/Migrate.csproj
    deployFiles: ./       # CELÝ source tree → /var/www/
    cache: true
  run:
    base: dotnet@9
    initCommands:
      - zsc execOnce ${appVersionId} --retryUntilSuccessful -- dotnet run --project Migrate/Migrate.csproj
    ports: [{port: 8080, httpSupport: true}]
    envVariables:
      ASPNETCORE_ENVIRONMENT: Development
      ...
      HOME: /home/zerops
    start: zsc noop --silent
```

**Dev deploy flow — agent's edit loop:**

1. Agent edituje code přes SSHFS mount `/var/www/uploaddev/`
2. `zerops_deploy targetService=uploaddev` — push celý tree včetně source, restore cache
3. Runtime container recyklován, new container:
   - `initCommands` běží dotnet run migrate (kompiluje on-the-fly)
   - `start: zsc noop --silent` — container idluje na 8080
4. Agent SSH → uploaddev, ručně `cd /var/www && dotnet run --project App.csproj` (on-the-fly compile + run)
5. Uvidí change → edit → restart dotnet run → loop

Iterace nejde přes deploy pipeline; deploy jen "wipne" container po velkých změnách (např. package.json).

### 2.4 Dotnet platform specifika (z docs)

- Build container má `dotnet@9` base: SDK + runtime + NuGet.
- `dotnet publish -c Release -o out` vytvoří **framework-dependent** artefact (runtime je už v dotnet@9 runtime image, takže nemusíme bundle-ovat).
- `cache: true` snapshot-uje NuGet cache — další build neresolve-uje packages znovu.
- Runtime container má dotnet@9 runtime, ale **ne SDK**. Proto dev setup musí runtime-restore + runtime-run, kterého se zbavuje přes `HOME=/home/zerops` a SSH interaktiv.

## 3. F3 — `deployFiles paths not found` false positive: root cause

### 3.1 Live repro (2026-04-24)

```go
// internal/ops/f3_repro_test.go (ephemeral, run then deleted)
yaml := `zerops:
  - setup: uploadstage
    build:
      base: dotnet@9
      buildCommands: [dotnet publish App.csproj -c Release -o out]
      deployFiles: [./out]
    run: {ports: [{port: 8080}], start: dotnet App.dll}
`
// ... writes to tempdir, runs ValidateZeropsYml

// OUTPUT (ověřeno):
// --- Source tree: /tmp/.../001 ---
//   zerops.yaml
// --- Validator output (role=stage) ---
// [0] deployFiles paths not found: ./out — these will be missing from the deploy artifact
// --- Validator output (role=<empty>) ---
// [0] deployFiles paths not found: ./out — these will be missing from the deploy artifact   ← ne-role-gated
// --- After mkdir out/ ---
// (no warnings)
```

**Observations:**
- Warning fires independentně na role (`""`, `"dev"`, `"stage"` — všechno hoří stejně).
- `mkdir out/` jako čistý adresář ho uspí. **Obsah NENÍ kontrolován** — jen stat, že path existuje.

### 3.2 Call chain přesně

```
deploy_ssh.go:125   ValidateZeropsYml(mountPath, setupName, serviceType)
   ↓
deploy_validate.go:22   ValidateZeropsYml(workingDir, targetHostname, serviceType, roles...)
   ↓
deploy_validate.go:87-100   // When cherry-picking (not "."), check each path exists.
   for _, df := range deployFiles {
       p := filepath.Join(workingDir, df)    // ← stat-uje HOST filesystem (SSHFS mount)
       if _, err := os.Stat(p); err != nil {
           missing = append(missing, df)
       }
   }
   if len(missing) > 0 {
       warnings = append(warnings, "deployFiles paths not found: ...")
   }
```

**Kontrola běží nad `workingDir`, což je `mountPath = /var/www/{sourceService}` — SSHFS mount na ZCP, jenž zrcadlí source tree target containeru.** Source tree nemá `./out`, protože `./out` je build output.

### 3.3 Root cause — sémantický rozpor s platformou

Platform docs (`../zerops-docs/apps/docs/content/alpine/how-to/build-pipeline.mdx:256-272`) definují:

> **deployFiles** — *REQUIRED.* Selects which files or folders will be deployed **after the build has successfully finished**.
>
> Determines files or folders **produced by your build**, which should be deployed to your runtime service containers.

`deployFiles` je definováno **nad post-build stromem** (source + buildCommands outputs). ZCP client-side check `os.Stat(filepath.Join(workingDir, df))` stat-uje **pre-build source tree**.

Tyto dva stromy se liší o buildCommands outputs. Pro:
- `deployFiles: [./src, ./package.json]` — oba cesty v source treeji. Check funguje coincidentálně.
- `deployFiles: [./out]` + buildCommands produces `./out` — cesta POST build. Check je kategoricky wrong.
- `deployFiles: [./dist]` (Vite, esbuild), `[./build]` (React), `[./target]` (Maven/Rust), `[./bin]` (Go, pokud `go build -o ./bin/app`) — všechno to samé.

Check se ptá wrong filesystem na wrong filesystem phase.

### 3.4 Duplicita se Zerops builder

Zerops builder v build containeru DĚLÁ vlastní post-build validaci `deployFiles`. Emituje `WARN: deployFiles paths not found: dist` (evidence v `internal/ops/build_logs_test.go:121`, `internal/tools/deploy_ssh_test.go:243`).

Post-`api-validation-plumbing.md` W4/W5, `FetchBuildWarnings` scope-uje logy by tag identity `zbuilder@<event.ID>` + facility `application` (fix I-LOG-2). Takže builder warning landuje v `DeployResult.BuildLogs` pouze pro tento konkrétní build, bez leaku z předchozích.

Client-side pre-flight check v `ValidateZeropsYml:87-100` je:
1. Chybně zaměřený (wrong filesystem phase)
2. Redundantní (builder validuje authoritatively)
3. Non-invalidating po úspěšném buildu (zůstává v `result.Warnings` forever)

### 3.5 Proč false positive přežívá úspěšný build

`deploy_poll.go:39-67` po `status=ACTIVE`:
- Nastaví `result.Status = "DEPLOYED"`
- Fetch `buildWarnings` z builderu → `result.BuildLogs`
- Volá `WaitSSHReady` → pokud timeout, **appenduje** na `result.Warnings` (neinvaliduje existing)
- **Pre-flight warnings z kroku 9 (ValidateZeropsYml) zůstanou intact**

Neexistuje žádný `result.Warnings = slices.DeleteFunc(...)` filter pro stále-platná pre-flight varování po úspěšném buildu.

### 3.6 Proč existing precedent `hasImplicitWebServer` nezachraňuje tento case

Commit `d961012` (2026-03-03) zavedl `hasImplicitWebServer()` helper a wrapped `run.start` / `run.ports` empty warnings za něj. **Jen tyto dvě** warnings jsou role-agnostic+runtime-class-aware. `deployFiles paths not found` check žádný takový wrap nedostal — pattern se neprováděl systematicky.

### 3.7 Tři fixové přístupy — doporučení

**Volba 1 — Delete the check (preferred, minimální)**:

Řádky `deploy_validate.go:86-100` odstranit. Rationale:
- Platform docs říkají jasně: deployFiles are post-build.
- Builder emituje authoritative warning pokud path opravdu missing.
- Client-side check nemá smysluplný informační signál — false-positive rate > 0 při jakémkoli builded language (.NET, Java, Rust, Go with `-o`, Node with bundlers).

Loss: V source-only deployments (no buildCommands) check zachytával typos ("deployFiles: `./nonexistend`" místo `./nonexistent`). Ale:
- Typo ve zdroji se projeví jako empty deploy artifact → builder warning → visible.
- Trade-off: jeden wasted build (~10-30s platform time) vs trvalý false-positive noise na každém .NET/bundled-JS/Go deploy.

**Volba 2 — Gate za prázdnost `buildCommands`**:

```go
// deploy_validate.go:86-100, nový gating
if len(entry.Build.BuildCommandsList()) == 0 && len(deployFiles) > 0 &&
    !slices.Contains(deployFiles, ".") && !slices.Contains(deployFiles, "./") {
    // cherry-pick check only makes sense for source-only deploys
    ...
}
```

Rationale: když je buildCommands non-empty, cherry-pickované paths jsou předpokládaně build outputs. Když je prázdné, cherry-pickované paths jsou source-tree paths a stat check má smysl.

Loss: Edge case — buildCommands použité pro něco jiného než build output (např. lint, test) + cherry-picked source paths. Test se přeskočí, typo se projeví jako empty artifact → builder warning.

**Volba 3 — Parse `buildCommands` pro inferred outputs (not recommended — maintenance burden)**:

Regex-match `-o`, `--output`, `--outdir` in commands, build inferred set, skip check for matches. Přesné, ale každý nový framework vyžaduje extension.

### 3.8 Test strategie pro volbu 1 (delete)

**Remove**: `TestValidateZeropsYml_DeployFiles` case `"cherry-picked deployFiles with missing paths"` (`deploy_validate_test.go`, cca řádek 285).

**Add**: `TestValidateZeropsYml_DeployFiles_BuildOutputIsNotPreFlightChecked` proving:
1. `deployFiles: [./out]` + `buildCommands: [...]` + `./out` neexistuje → NO warning.
2. `deployFiles: [./src]` + `buildCommands: []` + `./src` neexistuje → NO warning (delete = žádný check vůbec).

**Preserve** (žádná regrese):
- Tests checking `build.deployFiles is empty`
- Tests checking `run.deployFiles` is misplaced
- Tests checking `dev service should use deployFiles: [.]`
- Implicit-webserver tests

Update E2E scénář `internal/eval/scenarios/deploy-warnings-fresh-only.md` — pokud zde existuje dependency na pre-flight warning text.

## 4. F4 — dotnet recipe `./out` vs `./out/~`: root cause

### 4.1 Co se user's log stalo

User's zerops.yaml (zjednodušená ad-hoc varianta, NE recipe verbatim):

```yaml
- setup: prod
  build:
    base: dotnet@9
    buildCommands: [dotnet publish App.csproj -c Release -o out]
    deployFiles: [./out]
  run:
    base: dotnet@9
    start: dotnet ./out/App.dll    # user's first cut
```

Runtime po deployi:
- `/var/www/out/App.dll`
- `/var/www/out/wwwroot/` (když ASP.NET app má wwwroot s HTML/CSS)
- `/var/www/out/appsettings.json`, atd.

Process start:
- CWD při startu = `/var/www/`
- `dotnet ./out/App.dll` spustí dll z relative path
- ASP.NET 9's default: `ContentRootPath = Directory.GetCurrentDirectory()` = `/var/www`
- WebRootPath = ContentRootPath + "wwwroot" = `/var/www/wwwroot`
- **wwwroot/ je ale na `/var/www/out/wwwroot/`, ne `/var/www/wwwroot/`** → 404 na static assets

### 4.2 Tilde semantika per platform docs

From `../zerops-docs/apps/docs/content/nodejs/how-to/build-pipeline.mdx:338-362`:

> Zerops supports the `~` character as a wildcard for one or more folders in the path.
>
> `deployFiles: ./path/~/to/file.txt` — Deploys all `file.txt` files that are located in any path that begins with `/path/` and ends with `/to/`
>
> `deployFiles: ./path/to/~/` — Deploys all folders that are located in any path that begins with `/path/to/`
>
> **Example**: By default, `./src/assets/fonts` deploys to `/var/www/src/assets/fonts`, keeping the full path. Adding `~`, like `./src/assets/~fonts`, shortens it to `/var/www/fonts`

Practical interpretation (cross-checked s recipe body napříč 20 other recipes):

- `./PATH/~/` — "contents of `./PATH/` extracted do `/var/www/`" (strip `PATH/` z cesty)
- `./PATH/~SEGMENT` — "shortens path: `/var/www/{anything-before-PATH}/SEGMENT`"

### 4.3 Tilde pattern napříč recipes (21 hello-world recipes)

Recipes that use `.../~` extraction (all are STATIC or publish-style):

| Recipe | Path | Deployed to | Why |
|---|---|---|---|
| vue-static | `dist/~` | `/var/www/` | static serve |
| qwik-city-static | `dist/~` | `/var/www/` | static serve |
| angular-static | `dist/angular-hello-world/browser/~` | `/var/www/` | collapse Angular's nested output |
| react-router-static | `build/client/~` | `/var/www/` | static serve |
| solidstart-static | `.output/public/~` | `/var/www/` | static serve |
| gleam | `build/erlang-shipment/~` | `/var/www/` | extract erlang release |
| nuxt-static | `.output/public/~` | `/var/www/` | static serve |
| nextjs-static | `out/~` | `/var/www/` | static serve |

Recipes that preserve directory (no tilde):

| Recipe | deployFiles | Why it works |
|---|---|---|
| dotnet-hello-world (prod) | `./out` | multi-app layout (`out/app/`, `out/migrate/`); start references absolute `./out/app/App.dll` |
| laravel-minimal | `./` (full tree) | PHP doesn't build |
| bun-hello-world | (various setups) | mostly `./` for source-run |
| go-hello-world | `./bin/app` (single binary) | binary reference |

`internal/knowledge/bases/static.md` shrnuje "Framework Output Directories" s tildou — závazný pattern pro všechny static assets.

### 4.4 Root cause pro dotnet recipe

Dotnet recipe je designovaný pro **multi-app layout s migrací**. Tato architektura má dobré důvody:
- `out/app/` + `out/migrate/` — jasná separace app binary od migrate binary
- `initCommands` referencuje `./out/migrate/Migrate.dll` (absolute relative — stabilní)
- `start` referencuje `./out/app/App.dll` (absolute relative — stabilní)

**Ale**: tento design leak-uje do simpler use cases. LLM/uživatel, který má **single-app** ASP.NET aplikaci s wwwroot, zkopíruje recipe's `deployFiles: ./out` pattern. Publishes `./out/App.dll` + `./out/wwwroot/`. Runtime:

- `/var/www/out/App.dll` ✓
- `/var/www/out/wwwroot/` ✓
- `start: dotnet ./out/App.dll` → CWD=`/var/www`, ContentRootPath=`/var/www` ✗
- Static resources 404

### 4.5 ASP.NET ContentRootPath: důvod kolize

.NET 9's default `WebApplication.CreateBuilder(args)` sets ContentRootPath = `Directory.GetCurrentDirectory()`. Není to dll location; je to CWD.

**Three ways to fix at ASP.NET side (code change):**
1. `WebApplication.CreateBuilder(new WebApplicationOptions { ContentRootPath = AppContext.BaseDirectory })`
2. `builder.Host.UseContentRoot(AppContext.BaseDirectory)`
3. `chdir` to dll dir in Program.cs main

**Three ways to fix at zerops.yaml side (infra change):**
1. `deployFiles: ./out/~` + `start: dotnet App.dll` — extrahuje obsah do /var/www
2. `start: cd ./out && dotnet App.dll` — pokud platform supports cd prefix (risky — zsc může interpretovat command)
3. `initCommands: [cd /var/www/out/app]` — nefunguje, init nenastaví start-time CWD

**Volba 1 (zerops.yaml tilde extraction) je jediná robustní** pro use case "simple ASP.NET with wwwroot".

### 4.6 Návrh opravy recipe

Recipe by měl mít DVĚ PROD varianty (nebo default by měl být simple a migration variant separate):

**Simple single-app (DEFAULT)** — pro většinu use cases (CMS, web app, single API):

```yaml
- setup: prod
  build:
    base: dotnet@9
    buildCommands:
      - dotnet publish App.csproj -c Release -o out
    deployFiles:
      - ./out/~    # extract contents — /var/www/App.dll, /var/www/wwwroot/, etc.
    cache: true
  deploy:
    readinessCheck:
      httpGet: {port: 8080, path: /}
  run:
    base: dotnet@9
    ports: [{port: 8080, httpSupport: true}]
    envVariables:
      ASPNETCORE_ENVIRONMENT: Production
      ASPNETCORE_URLS: http://0.0.0.0:8080
    start: dotnet App.dll    # CWD=/var/www, ContentRootPath=/var/www, wwwroot=/var/www/wwwroot ✓
```

**With database migration (commented variant OR separate recipe)**:

```yaml
# --- WITH MIGRATION ---
# When you need a migration step before app start, use two-publish layout.
# Publish both to separate dirs under out/, keep out/ structure.
- setup: prod
  build:
    base: dotnet@9
    buildCommands:
      - dotnet publish App/App.csproj -c Release -o out/app
      - dotnet publish Migrate/Migrate.csproj -c Release -o out/migrate
    deployFiles:
      - ./out       # preserve out/ structure — /var/www/out/{app,migrate}/
    cache: true
  run:
    base: dotnet@9
    initCommands:
      - zsc execOnce ${appVersionId} --retryUntilSuccessful -- dotnet /var/www/out/migrate/Migrate.dll
    ports: [{port: 8080, httpSupport: true}]
    envVariables:
      ASPNETCORE_ENVIRONMENT: Production
      ASPNETCORE_URLS: http://0.0.0.0:8080
      # For wwwroot/ support with this layout, override ContentRootPath in Program.cs:
      #   WebApplication.CreateBuilder(new WebApplicationOptions
      #       { ContentRootPath = AppContext.BaseDirectory });
    # Start from /var/www/out/app dir so ContentRootPath=Directory.GetCurrentDirectory()
    # points to app root where wwwroot/ lives. zsc accepts bash-style CWD change.
    start: cd /var/www/out/app && dotnet App.dll
```

**DEV setup** — ponechat beze změny (`deployFiles: ./`, `start: zsc noop --silent`).

## 5. Souhrn — co děláme

### F3 — `deployFiles paths not found` false positive

**Root cause**: client-side ValidateZeropsYml stat-uje source-tree-paths, ale deployFiles je definováno nad post-build-paths (platform docs jsou autoritativní). Duplicate s builder-side check, nedoplňuje se, persistuje po successu.

**Fix**: **volba 1 — delete the check** (řádky 86-100 v `deploy_validate.go`). Builder validuje authoritative. Client-side je noise.

**TDD**: remove případ `"cherry-picked deployFiles with missing paths"`, add případy pinning build-output immunity.

**Impact**: jeden rozdíl — wasted build při typo-v-deployFiles, před-platform-cycle detection se ztratí. V LLM-only development má trade-off: LLM si umí wasted build spočítat (`BuildLogs` obsahuje builder warning), persistující false-positive je horší.

### F4 — dotnet recipe `./out` vs `./out/~`

**Root cause**: recipe designovaný pro multi-app layout (app + migrate), ale default struktura leak-uje do simpler cases a kolimuje s ASP.NET's ContentRootPath = CWD.

**Fix**: restructure recipe prod setup na **single-app-with-tilde** default + **multi-app-with-cd** variant. Edit přes `zcp sync pull/push`. Atom `develop-first-deploy-scaffold-yaml.md` rozšířit o tilde guidance.

**TDD**: corpus invariants (`internal/knowledge/recipes/*_test.go`) pinují tvar recipe. After edit, re-run sync pull to sync, verify atom reference.

**Impact**: default path pro new-dotnet-apps je clean. Multi-app variant dostane explicit `cd` guidance. Komentář v recipe varuje o ContentRootPath gotcha.

## 6. Cross-F3/F4 insights

### 6.1 Common pattern: "validate at wrong layer"

F3 (validator → source, skutečnost → post-build) a F4 (recipe → multi-app default, real use case → single-app) sdílejí vzorec: **kód/obsah validuje na jiné layeru než kde se fakticky projeví**. Fix je u obou ten samý: **posunout validaci blíž k autoritativnímu source** (platform builder pro F3; per-use-case recipe variants pro F4).

### 6.2 Invariant pro další podobné bugy

"ZCP neverifikuje to, co platforma sama autoritativně verifikuje" — generalizace fix-at-source. Promítá se do principu `api-validation-plumbing.md` W6 (delete duplicate validation). F3 je konkrétní instance: pre-flight filesystem check duplikuje builder filesystem check.

### 6.3 Recipe authorship reviewed-by-runtime

Dotnet recipe's multi-app assumption je "vendor pečlivě nastavil, LLM to teď kopíruje do simpler cases". Měl by existovat step v `zcprecipator3` flow, který **instrumentuje recipe use cases** — minimálně commentuje "if you don't have migration, drop Migrate block and use `./out/~`".
