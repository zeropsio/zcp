## Verdict per axis

### Axis 1 —  design (§7 Phase 0 step 3)
Verdict: NEEDS-REVISION
Findings:
1. The pin-density sketch says it asserts atom IDs are mentioned in `scenarios_test.go::wantIDs` or `corpus_coverage_test.go::MustContain`, but the code reads both whole files and uses `strings.Contains(haystack, atom.ID)`, so comments, rationale strings, and unrelated mentions count as pins (`plans/atom-corpus-hygiene-2026-04-26.md:765-800`, `plans/atom-corpus-hygiene-2026-04-26.md:807-810`). That is weaker than the stated design and weaker than the existing scenario helper contract, which specifically asserts atom IDs through `requireAtomIDsContain` / `requireAtomIDsExact` (`internal/workflow/scenarios_test.go:32-67`).
2. The sketch places `knownUnpinnedAtoms` in `corpus_coverage_test.go` and then includes `corpus_coverage_test.go` in the haystack (`plans/atom-corpus-hygiene-2026-04-26.md:774-800`). Without stripping the allowlist declaration, every allowlisted atom ID appears in the haystack by construction, so the same mechanism cannot distinguish a real pin from the allowlist entry itself.
3. The implementation should parse or otherwise isolate real pin sites rather than raw file substrings: `coverageFixture.MustContain` is the explicit phrase-pin surface (`internal/workflow/corpus_coverage_test.go:22-29`), while scenarios use `requireAtomIDsContain` / `requireAtomIDsExact` as the explicit atom-ID pin surface (`internal/workflow/scenarios_test.go:32-67`).

### Axis 3 — Recipe-atom scope leakage (R6)
Verdict: APPROVE
Findings:
1. The pin-density design uses `LoadAtomCorpus()` as its corpus source (`plans/atom-corpus-hygiene-2026-04-26.md:790-803`), and `LoadAtomCorpus()` reads `content.ReadAllAtoms()` (`internal/workflow/synthesize.go:448-472`), which is backed by `//go:embed atoms/*.md` and reads only the embedded `atoms` directory (`internal/content/content.go:19-20`, `internal/content/content.go:122-141`). That scopes the test to `internal/content/atoms/`, not `internal/recipe/`.
2. The §4.2 derivation script loops only over `internal/content/atoms/*.md`, so it excludes any recipe content regardless of where recipe markdown lives (`plans/atom-corpus-hygiene-2026-04-26.md:158-169`). This matches R6's mitigation that the new pin-density gate scope-filters to `internal/content/atoms/` only (`plans/atom-corpus-hygiene-2026-04-26.md:1450-1453`).
3. Recipe-active guidance is already documented as outside this atom corpus: the coverage test says `PhaseRecipeActive` is intentionally not covered because recipe authoring uses a separate section-parser pipeline and the atom corpus never owns recipe guidance (`internal/workflow/corpus_coverage_test.go:729-732`).

### Axis 5 — §6.3 fire-set generator approach
Verdict: NEEDS-REVISION
Findings:
1. A `cmd/atom_fire_audit` binary can import `workflow.LoadAtomCorpus`: `LoadAtomCorpus` is exported from `internal/workflow` (`internal/workflow/synthesize.go:448-454`), and the sketch imports `github.com/zeropsio/zcp/internal/workflow` from within the same module tree (`plans/atom-corpus-hygiene-2026-04-26.md:451-464`).
2. The sketch correctly uses `workflow.Synthesize`, so synthetic envelopes pass through the real envelope-axis and service-axis gates (`plans/atom-corpus-hygiene-2026-04-26.md:482-503`). In the real implementation, envelope axes are checked by `atomEnvelopeAxesMatch` (`internal/workflow/synthesize.go:157-187`) and service axes by `serviceSatisfiesAxes` (`internal/workflow/synthesize.go:228-262`).
3. The bootstrap synthetic generator omits services for every route/step pair (`plans/atom-corpus-hygiene-2026-04-26.md:578-590`), so any bootstrap atom with a service-scoped runtime axis cannot fire through that synthetic path because service-scoped atoms are skipped when no envelope service satisfies their axes (`internal/workflow/synthesize.go:61-74`). Current bootstrap atoms with runtime axes include `bootstrap-classic-plan-dynamic` and `bootstrap-classic-plan-static` (`internal/content/atoms/bootstrap-classic-plan-dynamic.md:1-9`, `internal/content/atoms/bootstrap-classic-plan-static.md:1-9`).
4. The plan says bootstrap-active should enumerate every `(Route, Step)` pair over `{discover, provision, generate, deploy, close}` (`plans/atom-corpus-hygiene-2026-04-26.md:448-449`), but atom frontmatter currently accepts only `discover`, `provision`, and `close` as valid `steps` enum values (`internal/workflow/atom.go:182-186`). The generator should either use the atom-valid set or explicitly mark `generate`/`deploy` as workflow-only impossible-for-atoms states.
5. The plan says develop-active generation includes `Trigger`, but the sketch's nested loops cover environment, mode, strategy, runtime, and deployed only; no trigger loop exists (`plans/atom-corpus-hygiene-2026-04-26.md:594-623`). Trigger is a service-scoped axis in the real synthesizer (`internal/workflow/synthesize.go:215-225`, `internal/workflow/synthesize.go:237-242`).
6. The sketch does not implement strategy-setup or export-active generation; it leaves them to "executor judgment" (`plans/atom-corpus-hygiene-2026-04-26.md:632-634`). That can falsely mark strategy atoms as marginal or dead, especially local strategy setup: `strategy-push-git-push-local` requires `phases: [strategy-setup]`, `strategies: [push-git]`, and `environments: [local]` (`internal/content/atoms/strategy-push-git-push-local.md:1-9`), while existing coverage fixtures cover strategy setup only in container environment (`internal/workflow/corpus_coverage_test.go:653-709`).
7. The sketch does not generate `develop-closed-auto` synthetic envelopes even though the phase set includes it (`plans/atom-corpus-hygiene-2026-04-26.md:445-449`, `plans/atom-corpus-hygiene-2026-04-26.md:594-635`). The current atom `develop-closed-auto` is pinned by existing fixtures (`internal/content/atoms/develop-closed-auto.md:1-8`, `internal/workflow/corpus_coverage_test.go:272-285`), but a fire-set generator advertised as exhaustive should generate that phase directly.
8. The sketch's develop generator uses only a single service per envelope (`plans/atom-corpus-hygiene-2026-04-26.md:617-623`), so it misses mixed deploy-state and multi-service cases. That matters because `envelopeDeployStates` matches if any bootstrapped service has the wanted state (`internal/workflow/synthesize.go:189-212`), and the plan's own Axis J risk names first-deploy atoms firing in mixed envelopes as a key looseness problem (`plans/atom-corpus-hygiene-2026-04-26.md:301-304`).
9. There is no current `envelopeRuntimes` atom axis: valid frontmatter keys include `runtimes` and `envelopeDeployStates` but not `envelopeRuntimes` (`internal/workflow/atom.go:84-103`). If Phase 0 expects envelope-scoped runtime generation, the axis does not exist yet and should not be implied by the fire-set design.
