## Findings (Phase 0 → Phase 1 / sidecar work)

### F0-DEAD-1 — `bootstrap-recipe-close` is content-bug-surfaced, NOT axis-dead

This atom's body line 25-26 contains the literal `strategies={hostname:value}`
which `internal/workflow/synthesize.go::isPlaceholderToken` accepts as a
placeholder token (no whitespace, no braces, no quotes inside —
colons are allowed). The token is not in the `allowedSurvivingPlaceholders`
allowlist nor in the `replacer` substitutions, so `findUnknownPlaceholder`
returns it as an unknown placeholder and `Synthesize` errors out for
**every** envelope where this atom would otherwise fire (recipe/close
shape).

10 envelopes failed Synthesize during this fire-audit run — all 10
errors had this exact root cause. Confirmed by stderr log of
`atom_fire_audit`:

```
bootstrap/local/recipe/close/svc-implicit-webserver: atom bootstrap-recipe-close: unknown placeholder "{hostname:value}"
bootstrap/container/recipe/close/no-svc:             atom bootstrap-recipe-close: unknown placeholder "{hostname:value}"
bootstrap/local/recipe/close/no-svc:                 atom bootstrap-recipe-close: unknown placeholder "{hostname:value}"
bootstrap/local/recipe/close/svc-dynamic:            atom bootstrap-recipe-close: unknown placeholder "{hostname:value}"
bootstrap/local/recipe/close/svc-managed:            atom bootstrap-recipe-close: unknown placeholder "{hostname:value}"
bootstrap/container/recipe/close/svc-dynamic:        atom bootstrap-recipe-close: unknown placeholder "{hostname:value}"
bootstrap/container/recipe/close/svc-static:         atom bootstrap-recipe-close: unknown placeholder "{hostname:value}"
bootstrap/container/recipe/close/svc-implicit-webserver: atom bootstrap-recipe-close: unknown placeholder "{hostname:value}"
bootstrap/container/recipe/close/svc-managed:        atom bootstrap-recipe-close: unknown placeholder "{hostname:value}"
bootstrap/local/recipe/close/svc-static:             atom bootstrap-recipe-close: unknown placeholder "{hostname:value}"
```

**Production impact:** every real recipe/close envelope errors during
status rendering, dropping all bootstrap-active guidance for that
phase. The corpus test suite is green only because no test fixture
exercises a recipe/close envelope (verified: `grep -nE "recipe.*close"
internal/workflow/scenarios_test.go internal/workflow/corpus_coverage_test.go`
returns no matches).

**Disposition:** NOT a Phase 1 dead-atom candidate. Sidecar fix
needed — change `strategies={hostname:value}` to a syntax that
escapes the placeholder detection (e.g. `strategies={"hostname":"value"}`
in proper JSON form, since `isPlaceholderToken` rejects tokens
containing `"`). Track between Phase 0 EXIT and Phase 1 ENTRY.

## Caveats (per §6.3 + Codex round 1 axis 5)

1. The synthetic envelope generator does NOT walk
   `corpus_coverage_test.go::coverageFixtures()` — _test.go is not
   importable from `cmd/`. The Cartesian product + multi-service
   mixed envelope is comprehensive enough for dead-atom detection
   (every atom-axis combination is exercised), but specific multi-
   service shapes from coverage fixtures (e.g. two-runtime-pair
   develop) aren't generated separately. Phase 1 Codex round must
   walk `ComputeEnvelope` to confirm any DEAD candidate isn't
   merely a fixture-shape miss (per R3 mitigation).

2. The 10-envelope error rate is bounded entirely by F0-DEAD-1
   (bootstrap-recipe-close placeholder bug). Once that's fixed,
   re-run yields 0 errors.

3. Fire-set count is a coarse signal. An atom firing on 4610
   envelopes isn't necessarily worse than one firing on 10 — count
   reflects the BREADTH of the axis filter, not whether the
   content is RELEVANT to those envelopes (Axis J distinction).
   Phase 7 task-relevance scoring resolves this.
