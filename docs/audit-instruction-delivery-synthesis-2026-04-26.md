### Authoritative state per atom-axis combination

`internal/workflow/corpus_coverage_test.go` pins non-empty Synthesize output for
each `(Phase, Environment)` pair the production code emits. **What it does not
pin** is the *shape* of the matched set — i.e., which atom IDs fire for a given
envelope. With the new `MatchedRender.AtomID` (post-C2), tests can assert atom
sets, but the migration of `scenarios_test.go` from `SynthesizeBodies` →
`MatchedRender` (Phase 4.2b of `plans/test-suite-fixes-2026-04-25.md`) hasn't
happened yet — substring matching on joined bodies is still the primary
assertion shape there.

## 3. Gaps that slipped through phase1–5

Each gap names: surface (file:line), drift mechanism, repro scenario, and
whether it was observed live during this audit.

### G1 — `zerops_recipe` tool description omits `record-fragment` action *(Codex; not observed live this run)*

- **Surface.** `internal/recipe/handlers.go:164` — MCP description names a
  subset of actions; `record-fragment` is in the schema (`:107`) and dispatcher
  (`:273`) but absent from the description.
- **Drift mechanism.** `description_drift_test.go:48` scans `../tools/*.go`
  only. `internal/recipe/handlers.go` is in a separate package, so the lint
  does not see it. `annotations_test.go:69-88` lists tools to verify;
  `zerops_recipe` is not in the list.
- **Repro.** Recipe v3 active; agent reads MCP init tool list, sees the
  abridged description, never calls `record-fragment`. Recipe halts on the
  first fragment-required step.
- **Fix.** Extend description-drift scan to the v3 recipe package; include
  `zerops_recipe` in the annotations completeness test.

### G2 — v3 recipe sessions are invisible to MCP `Instructions` AND `zerops_workflow status` *(Codex; structural)*

- **Surface.** `internal/server/server.go:83` builds `RuntimeContext` from
  v2 workflow registry only; `internal/server/instructions.go:72-99`
  enumerates v2 recipe + v2 bootstrap + v2 work session, never the v3
  in-memory recipe store (`internal/recipe/handlers.go:17-28`).
  `internal/tools/workflow.go:234-242` (status routing) considers only v2
  recipe / v2 bootstrap / develop lifecycle.
- **Drift mechanism.** Recovery primitive (`zerops_workflow action="status"`)
  is the contract. After context compaction, an agent in a v3 recipe session
  receives an idle/develop response — not the recipe state — and cannot
  re-orient. Storage divergence between v2 (registry on disk) and v3
  (in-memory) means StateHint can never see v3.
- **Repro.** Start `zerops_recipe action="start" slug="..."`; force compaction;
  call `zerops_workflow action="status"`. Response will not name the v3
  recipe session.
- **Fix direction.** Recipe v3's `recipeStore` must register with the same
  registry workflow looks at, OR the canonical recovery doctrine must extend
  to "call `zerops_recipe action=status` first if recipe-active, otherwise
  `zerops_workflow action=status`" (the CLAUDE.md template already says this
  — but the MCP `Instructions` `StateHint` cannot point to it without seeing
  the v3 store).

### G3 — Corrupt work-session file silently defeats compaction recovery *(Codex; structural)*

- **Surface.** `internal/workflow/work_session.go:122-124` returns
  `unmarshal work session` on bad JSON; `compute_envelope.go:61-70` propagates;
  `internal/tools/workflow.go:759-765` bubbles into `convertError` → leaf
  payload with empty suggestion. `internal/server/instructions.go:92`
  silently drops the same error (LoadWorkSession discarded `err`).
- **Drift mechanism.** Recovery primitive depends on a parseable file; both
  the init hint AND the status call fail differently and neither tells the
  agent the file is corrupt or how to reset it.
- **Repro.** `develop-active`, corrupt `.zcp/state/work/{pid}.json`, call
  `zerops_workflow action="status"`. Get error "unmarshal work session" with
  empty `suggestion`.
- **Fix direction.** `convertError` for corrupt-state errors should attach
  a recovery suggestion citing `action="reset"` (already exists at
  `workflow.go:377-410`); init's silent drop should at minimum log a warning.

### G4 — Orphan incomplete meta + externally-deleted service drops resume affordance *(Codex; structural)*

- **Surface.** `internal/workflow/compute_envelope.go:125-140` derives
  `IdleIncomplete` from `ServiceSnapshot.Resumable`, but
  `compute_envelope.go:176-184,216-221` only builds snapshots by iterating
  *live* platform services. If the service was deleted externally, the
  ServiceMeta on disk still says incomplete, but no snapshot exists →
  `Resumable=false` everywhere → `IdleIncomplete` cannot fire.
  `corpus_coverage_test.go:42-90` covers idle empty/bootstrapped/adopt
  but not idle-incomplete.
- **Repro.** `bootstrap-active` interrupted after partial meta; live service
  deleted via dashboard; call `zerops_workflow action="status"`. Idle
  envelope routes to "start bootstrap" (empty) or "adopt" (visible
  unmanaged), never to "clean up the orphan meta".
- **Fix direction.** ComputeEnvelope must check disk metas independently of
  live snapshots; emit a synthetic snapshot or a separate `OrphanMetas`
  envelope field. Add corpus_coverage idle-incomplete fixture.

### G5 — In headless mode from non-init'd directories, the LLM gets ZERO workflow doctrine *(observed live in P0)*

- **Surface.** Container env → MCP `Instructions` is empty when no sessions
  (`server.go:76-83` — AdoptionNote is local-env-only; `ComposeStateHint`
  empty when no sessions). CLAUDE.md auto-discovery requires `<cwd>/CLAUDE.md`
  → only present where `zcp init` was run. `--bare` (or running from `~`,
  `/tmp`, an audit dir, an unrelated project) means the LLM has only tool
  descriptions.
- **Live observation.** P0 (`/home/zerops/audit-2026-04-26/`, no CLAUDE.md):
  prompt "what is this project, please use the zerops MCP tools to investigate".
  LLM correctly discovered services via `zerops_discover` and described
  shape — but it **never mentioned** the three entry points, the canonical
  `status` recovery call, SSHFS mount semantics, or any operational next
  step. Tool descriptions alone do not teach workflow doctrine.
- **Drift mechanism.** Workflow doctrine lives in the CLAUDE.md template;
  it is never delivered through MCP. Running headless in a non-init'd dir
  is a real eval / scripted-automation entry point and produces a doctrine-
  starved LLM.
- **Repro.** `cd /tmp && claude --print "..."` in a project bound to a Zerops
  MCP. The LLM has no sense of "should I start a workflow first" or
  "after compaction call status".
- **Fix direction.** Either (a) put a *minimal* doctrine pointer in the MCP
  `Instructions` field always (one line: "Three entry points: develop,
  bootstrap, recipe; recovery via `zerops_workflow action=status`"), or
  (b) document that `zcp init` is mandatory in any working directory
  before headless use.

### G6 — CLAUDE.md template's `scope=["appdev"]` example is taken literally *(observed live in P3)*

- **Surface.** `internal/content/templates/claude_shared.md:14` —
  ```
   zerops_workflow action="start" workflow="develop" \
     intent="<one-line proposal>" scope=["appdev"]
  ```
  `appdev` is a hardcoded literal, not a `{hostname}` placeholder.
- **Live observation.** P3 (`/var/www/`, fresh CLAUDE.md, no `appdev`
  service in project): prompt "I want to add a /healthz health endpoint on
  appdev and verify it works." LLM called `zerops_workflow action="start"
  workflow="develop" intent="..." scope=["appdev"]` — verbatim from the
  template. The user's prompt also said "appdev", so this particular case
  matched, but only by coincidence.
- **Drift mechanism.** `appdev` is the hostname used in the eval-zcp
  fixture project. Real users in projects with different hostnames will get
  the LLM trying `scope=["appdev"]` regardless of what services actually
  exist. The template doesn't say "replace with your service hostname".
- **Repro.** `zcp init` a project with runtime hostname `webapp` (no
  `appdev`); ask LLM to start develop. It will use `scope=["appdev"]` and
  hit `PREREQUISITE_MISSING`.
- **Fix direction.** Template should use a placeholder syntax that's
  obviously a placeholder (e.g., `scope=["<hostname>"]` or `scope=["YOUR_SERVICE"]`),
  or omit the example and direct the agent to call `zerops_discover` first.

### G7 — Spec-promised `GhostServices` flagging is not implemented *(spec drift)*

- **Surface.** `docs/spec-scenarios.md:357-361` (§6.5 "Service deleted
  externally mid-work-session") promises `ComputeEnvelope` flags missing
  services as `GhostServices=[<names>]` and `BuildPlan` returns
  "close current develop, services no longer exist".
- **Verification.** `grep -rE "GhostService|MissingService" internal/workflow/`
  yields nothing in production code; only `recipe_validate_test.go:185`
  mentions `MissingServices` (a different concept).
- **Drift mechanism.** Spec scenario reads as if implemented; corpus_coverage
  doesn't test it; an LLM that runs into this case gets the regular
  develop-active envelope, not the ghost-service plan.
- **Fix direction.** Either implement (small change in
  `compute_envelope.go` to diff `WorkSession.Services` against live
  `env.Services`, attach a new `GhostServices []string` field) or remove
  the spec promise. Pin with a unit test.

### G8 — Per-service `/var/www/{hostname}/CLAUDE.md` referenced but ungoverned *(structural)*

- **Surface.** `internal/content/templates/claude_container.md:10` —
  *"Per-service rules (reload behaviour, start commands, asset pipeline)
  live at `/var/www/{hostname}/CLAUDE.md` — read before editing."*
- **Drift mechanism.** No template, no generator, no lint. The agent reads
  these files and acts on them, but their content is whatever a developer
  happens to write — including drift from atom doctrine, stale advice
  from a previous recipe iteration, or contradicting platform invariants.
- **Repro.** Place a `/var/www/appdev/CLAUDE.md` file that says "always
  call `zerops_subdomain action=enable` after deploy" (the C7-corrected
  drift wording). The drift lint does not scan it. The LLM follows it.
- **Fix direction.** Either (a) make per-service CLAUDE.md a generated
  artifact (generated at bootstrap close, owned by ZCP) so its body is
  governed; or (b) extend `description_drift_test.go` to scan
  `e2e/fixtures/**/CLAUDE.md` and document the expectation; or (c) accept
  it as user-territory and stop linking to it from claude_container.md
  template — let users decide.

### G9 — Stale CLAUDE.md drift on long-lived containers *(observed live)*

- **Surface.** This audit found `/var/www/CLAUDE.md` on the eval-zcp
  container had the pre-phase3 wording (sha `7950a88e…`, last modified
  2026-04-25) — *"Direct tools skip the workflow … auto-apply without a
  deploy cycle"* — exactly two forbidden patterns the C7 lint catches at
  build time.
- **Drift mechanism.** `zcp init` writes the template once. The drift lint
  is build-time and would refuse to release with that wording today, but
  the *deployed* CLAUDE.md is stamped at init-time; nothing re-renders on
  subsequent server starts. A long-lived container holds the snapshot
  from whichever zcp version last ran `zcp init` there.
- **Live observation.** Refreshed via `cd /var/www && zcp init`; sha
  changed to `03dfdfc3…`; current wording matches `claude_shared.md`.
- **Fix direction.** On `zcp serve` start in container env, idempotently
  re-render CLAUDE.md when the embedded template SHA differs from the
  on-disk SHA. Atomic write through the existing
  `<!-- ZCP:BEGIN --> / END -->` markers preserves user additions
  (REFLOG, custom sections).

### G10 — `convertError` still doesn't carry envelope or plan *(F4-residual; observed live in P5)*

- **Surface.** `internal/tools/convert.go:43-75` returns
  `{code, error, suggestion?, apiCode?, diagnostic?, apiMeta?}` with no
  envelope. P5 confirmed wire shape:
  ```json
  {"code":"SERVICE_NOT_FOUND","error":"Service 'doesnotexistdev' not found",
   "suggestion":"Available services: ..."}
  ```
- **Why it still matters after spec phase5.** P4 says terse mutation
  responses are fine because `status` is the recovery surface. But a
  failed `zerops_deploy` mid-develop returns this leaf error; the LLM
  has no lifecycle context until it makes a *second* round-trip to
  `status`. P3 confirmed the same shape on `PREREQUISITE_MISSING` — the
  LLM had to ask the user instead of acting on envelope state it never
  saw.
- **Fix direction.** Either (a) attach envelope+plan to errors when the
  handler already has the engine in scope (cheap — one ComputeEnvelope
  call), or (b) document explicitly in CLAUDE.md template that "after
  any tool error, call `action=status` to re-derive lifecycle context".
  Today neither is true.

### G11 — Free-form `next` strings still scattered in handler responses *(F4-T1 residual)*

- **Surface.** `workflow.go:361,409,810` (reset, close), `workflow_strategy.go:33,166,176,217`
  (strategy update + list). Each is a hand-rolled string instructing the
  LLM what to call next.
- **Drift mechanism.** These are unlinted Go string literals that overlap
  semantically with what `BuildPlan.Primary.Label/Args` would produce.
  When the typed Plan rationale is updated, these strings stay stale.
- **Fix direction.** Replace each with a typed `Plan` (already exists);
  delete the free-form strings. Pin with a lint that forbids `"next":`
  keys in `internal/tools/workflow*.go` JSON outputs.

### G12 — `dispatch-brief-atom` action bypasses engine-state guard *(Codex flagged)*

- **Surface.** `internal/tools/workflow.go:160,166` (action routing),
  `:497-527` (handler).
- **Drift mechanism.** Atoms are the structured runtime guidance corpus;
  fetching one by ID outside an active session means the agent can pull
  whatever atom prose it wants regardless of phase. It bypasses the
  envelope-driven filtering that's the corpus's safety property.
- **Fix direction.** Either gate the action on an active recipe/develop
  session, or document explicitly that the action is intentionally
  unguarded and used by sub-agents that don't carry the engine context.

### 4.3 New findings from live exercises (G13–G16)

#### G13 — Develop briefing payload size approaches the 32 KB MCP cap *(observed live in E2)*

- **Surface.** `internal/tools/workflow_develop.go:151-173` →
  `RenderStatus(Response{Envelope, Guidance, Plan})` → `textResult(...)`.
- **Live observation.** E2's develop-start tool result was **29 352 bytes** of
  rendered markdown for a *single-service simple-mode first-deploy* envelope.
  Audit's prior T2 noted the 32 KB cap awareness was recipe-only — but the
  develop briefing (single service, simplest mode) already lands at 91 % of
  the cap. A multi-service standard-mode first-deploy envelope (atoms fire
  per-service via `MatchedRender`) would multiply this.
- **Drift mechanism.** No size gate exists for runtime atoms.
  `corpus_coverage_test.go` only verifies non-empty Synthesize output; size is
  unbounded. The dispatch-brief-atom escape hatch exists for recipes but is
  not wired to runtime briefings.
- **Repro.** Bootstrap two services in standard mode (dev+stage pair × 2);
  start develop on both; observe briefing size approach or exceed 32 KB.
- **Fix direction.** Add a size gate in `corpus_coverage_test.go` for
  representative envelope shapes (single dev, dev+stage, multi-service
  standard). Over-cap triggers either chunked delivery (existing
  dispatch-brief-atom path) or a build-time error.

#### G14 — Three error wire shapes coexist; LLM must guess which it got *(observed live in E3 vs P5)*

- **Surface.** Three distinct error-response wire formats reach the LLM today:
  1. **Plain text** (`internal/tools/convert.go:46-49`) — for non-PlatformError
     errors. LLM sees raw error string.
  2. **`{code, error, suggestion?, apiCode?, diagnostic?, apiMeta?}` JSON**
     (`convert.go:51-62`) — `convertError` shape for PlatformError. Observed
     in P5 (SERVICE_NOT_FOUND from deploy) and P3 (PREREQUISITE_MISSING from
     workflow start).
  3. **`{passed: false, checks: [{name, status, detail}], summary}` JSON**
     — verify/preflight shape. Observed in E3 (zerops_yml_exists check fail).
- **Drift mechanism.** Each shape has its own field names. The LLM has to
  pattern-match on whichever it gets to extract the failure reason. Shape (3)
  has no `code`/`suggestion`/`apiMeta`; shape (2) has no `checks`; shape (1)
  has nothing structured. F4-T1 was about free-form `next` strings; this is
  the same problem class for errors.
- **Drift evidence (E3 prose).** The LLM's response correctly noted: *"No
  suggestion/next-action field; the model has to infer the fix"* — proving
  the LLM has to reason about each shape on its own.
- **Fix direction.** Either (a) normalize all error shapes to a single
  `{code, error, suggestion?, checks?, ...}` envelope, or (b) document each
  shape in CLAUDE.md template so the LLM knows the field grammar by tool name.

#### G15 — Bootstrap close emits hand-rolled markdown that overlaps atoms + CLAUDE.md *(observed live in E1)*

- **Surface.** E1's close-step response `message` field carried (verbatim from
  the wire):
  - `## Services` listing (factual, OK)
  - `## Deploy Model (read before developing)` — 4 bullets restating
    "Deploy = new container", "Code on SSHFS mount", "prepareCommands need sudo",
    "Build ≠ Run".
  - `## What's Next?` — A/B/C menu pointing to develop / recipe / scale.
- **Drift mechanism.** All 4 "Deploy Model" bullets are also present in atom
  bodies (e.g. `develop-platform-rules-container`) AND in the CLAUDE.md template
  (`claude_container.md:6-7`, `claude_shared.md:48-52`). Three sources for the
  same facts. The bootstrap close text is hand-rolled Go (likely
  `internal/workflow/bootstrap_outputs.go` or sibling), not atom-driven.
  The "What's Next?" menu duplicates `BuildPlan`'s output.
- **Repro.** Grep for the verbatim string "Deploy = new container" across
  `internal/workflow/`, `internal/content/atoms/`, `internal/content/templates/`
  — multiple sources will hit.
- **Fix direction.** Replace bootstrap close `message` with a typed Plan
  (the Primary action is "start develop"; rationale is the existing
  `## What's Next?` content) and route through `Synthesize` for the platform
  invariants. Delete the hand-rolled markdown.

#### G16 — Deploy preflight searches CWD instead of per-service path *(observed live in E3; ergonomics)*

- **Surface.** E3 deploy error: *"zerops.yaml not found in `/var/www`"*. CWD
  was `/var/www`; service was `probe`; SSHFS mount is `/var/www/probe/`.
  The preflight searched the *parent* directory, not the per-service mount.
- **Drift mechanism.** `claude_container.md:10` says per-service code lives
  at `/var/www/{hostname}/`. The deploy handler's `workingDir` defaults to
  CWD unless explicitly set. CLAUDE.md doesn't say "cd to the service mount
  before deploy" or "pass workingDir explicitly". The agent has to infer
  the right pattern from prose.
- **Live observation.** The LLM's E3 response noted: *"create `zerops.yaml`,
  likely under `/var/www/probe/` given the SSHFS layout"* — proving the
  agent CAN figure it out from CLAUDE.md prose, but only after the error.
- **Fix direction.** Either (a) deploy handler defaults `workingDir` to
  `/var/www/{targetService}/` when one isn't passed and the tool runs in
  container env, or (b) the workflow develop briefing surfaces the explicit
  `workingDir=...` argument in the Plan.Primary so the LLM doesn't infer.

### 4.4 What was confirmed about phase1–5 fixes (positive validation)

- **Bootstrap end-to-end works on the live API** — E1's 8-call sequence

**Codex review pass** caught two real bugs in the initial fix shape — both addressed:

- `RefreshClaudeMD` had a slice-low-greater-than-high panic when the
  on-disk file's `<!-- ZCP:END -->` marker appeared before
  `<!-- ZCP:BEGIN -->` (or the end was missing entirely). The end-marker
  search now scopes to `text[beginIdx+len(begin):]`, so reversed /
  begin-only / missing-end inputs fall through to the no-op path
  rather than crashing MCP startup. Tests
  `TestRefreshClaudeMD_ReversedMarkers_NoCrash` and
  `TestRefreshClaudeMD_BeginOnly_NoCrash` pin both pathological inputs.
- `findAndParseZeropsYml` originally treated "mount path probe failure"
  the same as "file not present" — both fell through to the project-root
  yaml. Two refinements: (a) when the per-service mount has a present
  but invalid `zerops.yaml`, the parse error returns immediately
  (`TestDeployPreFlight_InvalidMountYaml_DoesNotFallBackToRoot`);
  (b) tri-state probe (`mountStateAbsent` / `mountStateNoYaml` /
  `mountStatePresent` + `probeErr` path) — any non-IsNotExist stat
  failure (permission denied, stale SSHFS, non-directory) surfaces
  immediately rather than shadowing into a root-yaml validation
  (`TestDeployPreFlight_MountProbeError_DoesNotFallBackToRoot`).

Findings still open after this pass:

- **G14** (three error wire shapes — plain text vs. `convertError` JSON vs. preflight `{passed, checks[], summary}`). Documenting the grammar in CLAUDE.md or normalizing the shapes is a substantive design call beyond a tight-fix sweep.
- **G10** (errors carry no envelope). The G3 wrapper preserves typed errors when present, but the broader "every error response carries the live envelope" change is a structural refactor better done as its own plan.
- **G4** (orphan incomplete meta + externally-deleted service drops resume affordance). Needs `ComputeEnvelope` to read disk metas independently of live snapshots — design decision.
- **G5** (headless from non-init'd directory gets no doctrine). Adding any prose to `Instructions` field violates `TestBuildInstructions_NoStaticRulesLeak`. The honest fix is operator documentation (mark `zcp init` as mandatory before headless use) rather than weakening the test.
- **G8** (per-service `/var/www/{hostname}/CLAUDE.md` referenced but ungoverned). Policy decision: govern via a generated artifact, extend lint scan, or remove the link. Not addressed.
- **Recipe-scope skips by user request**: G1 (`zerops_recipe` description gap), G2 (v3 recipe sessions invisible to status), G12 (`dispatch-brief-atom` engine guard).
- **G13 corpus trim**: The new size gate explicitly allowlists two
  representative envelopes whose synthesized output already exceeds the
  32 KB MCP tool-response cap on the current corpus —
  `develop_first_deploy_standard_container` (40 228 bytes) and
  `develop_first_deploy_implicit_webserver_standard` (43 447 bytes).
  The companion `KnownOverflows_StillOverflow` test enforces that the
  allowlist can only shrink (any future trim that brings either fixture
  under the ceiling fails the companion test and forces removal from
  the allowlist). Codex's final-pass review correctly flagged that the
  underlying LIVE failure mode (the runtime status path can still
  return truncated or oversized payloads for those exact envelope
  shapes) is preserved until a corpus-trim plan lands. The decision
  to ship the gate-with-allowlist follows `plans/test-suite-fixes-2026-04-25.md`
  §2.2's documented pattern: enforce forward, defer the substantive
  per-atom content work to its own plan rather than block this sweep.
