# Deno Hello World on Zerops



## Keywords
deno, typescript, javascript, ubuntu, zerops.yml, permissions

## TL;DR
Deno runtime. REQUIRES `os: ubuntu` (not available on Alpine). Use `deno.jsonc`. Bind `0.0.0.0` via `Deno.serve`.

### Base Image

Includes Deno runtime.

**OS**: `os: ubuntu` REQUIRED (not available on Alpine).

### Build Procedure

1. Set `build.base: deno@latest`, `build.os: ubuntu`
2. Permissions: `--allow-net --allow-env` minimum
3. Use `deno.jsonc`, not `deno.json`

### Binding

`Deno.serve({hostname: "0.0.0.0"}, handler)`

### Build Caching

Run `deno cache main.ts` in buildCommands to pre-download dependencies. Ensures deployments are deterministic.
Cache: deps in `~/.cache/deno` (auto-cached).

### Resource Requirements

**Dev** (cache/compile on container): `minRam: 0.5` — `deno cache` moderate peak.
**Stage/Prod**: `minRam: 0.25` — Deno runtime lightweight.

### Deploy Patterns

**Dev deploy**: `deployFiles: [.]`, `start: zsc noop --silent` (idle container -- agent starts `deno run --allow-net --allow-env main.ts` manually via SSH for iteration)
**Prod deploy**: `deployFiles: [.]`, `start: deno run --allow-net --allow-env main.ts`
For compiled binaries: `deno compile --output app main.ts` -> `deployFiles: app`, `start: ./app`

## zerops.yml

> Reference implementation — learn the patterns, adapt to your project.

```yaml
zerops:
  # Production setup — compile TypeScript to self-contained binaries,
  # deploy minimal artifacts. Matching Deno 2 in build and runtime
  # ensures compiled binaries link against the same runtime internals.
  - setup: prod
    build:
      base: deno@2.0.0

      # Compile app and migration to self-contained binaries.
      # --allow-net: HTTP server + PostgreSQL. --allow-env: reads
      # env vars. The binary embeds the Deno runtime — runs without
      # deno installed on the host. --lock enforces the lock file
      # for reproducible builds across build containers.
      buildCommands:
        - deno compile --allow-net --allow-env --lock deno.lock
            --output app src/main.ts
        - deno compile --allow-net --allow-env --lock deno.lock
            --output migrate src/migrate.ts

      deployFiles:
        - ./app
        - ./migrate

      # cache: true snapshots the global Deno module cache
      # (~/.cache/deno) so dependencies are not re-downloaded
      # on subsequent builds. Use this instead of cache: [folder]
      # because Deno's cache lives outside the build directory.
      cache: true

    # Readiness check: verifies containers are serving traffic
    # before the project balancer routes requests to them.
    deploy:
      readinessCheck:
        httpGet:
          port: 8000
          path: /

    run:
      base: deno@2.0.0

      # Run migration once per deploy (execOnce). In initCommands —
      # not buildCommands — so migration and code deploy atomically.
      # execOnce ensures exactly one container runs the migration;
      # others wait, preventing race conditions in multi-container
      # deployments. --retryUntilSuccessful handles the brief window
      # on fresh project imports where the database is still starting
      # up even after priority 10 ordering.
      initCommands:
        - zsc execOnce ${appVersionId} --retryUntilSuccessful -- ./migrate

      ports:
        - port: 8000
          httpSupport: true

      envVariables:
        PORT: "8000"
        DB_NAME: db
        # Referencing variables: ${hostname_key} resolves at runtime
        # to the value generated by the 'db' PostgreSQL service.
        DB_HOST: ${db_hostname}
        DB_PORT: ${db_port}
        DB_USER: ${db_user}
        DB_PASS: ${db_password}

      start: ./app

  # Development setup — deploy full source code so a developer (or
  # AI agent) can SSH in and start working immediately. Zerops
  # prepares the workspace and gets out of the way.
  - setup: dev
    build:
      base: deno@2.0.0

      envVariables:
        # Redirect Deno module cache into the project tree so it
        # ships with deployFiles. The runtime container then works
        # fully offline — no re-downloads needed after deploy.
        DENO_DIR: ./.deno_cache

      # Pre-warm the cache for app and migration source files.
      # deployFiles: ./ ships the populated .deno_cache directory
      # to the runtime container alongside the source.
      buildCommands:
        - deno cache src/main.ts src/migrate.ts

      deployFiles:
        - ./

      # Cache the redirected DENO_DIR between builds.
      cache:
        - .deno_cache

    run:
      base: deno@2.0.0

      initCommands:
        # Migration runs once per deploy — database schema is ready
        # when the developer SSHs in. Offline because DENO_DIR is
        # deployed with the source (no network needed).
        # --retryUntilSuccessful handles fresh project imports where
        # the database port may not be open yet.
        - zsc execOnce ${appVersionId} --retryUntilSuccessful
            -- deno run --allow-net --allow-env src/migrate.ts

      ports:
        - port: 8000
          httpSupport: true

      envVariables:
        PORT: "8000"
        DB_NAME: db
        DB_HOST: ${db_hostname}
        DB_PORT: ${db_port}
        DB_USER: ${db_user}
        DB_PASS: ${db_password}
        # Point Deno to the shipped cache — all imports resolve
        # offline without downloading from deno.land.
        DENO_DIR: /var/www/.deno_cache

      # Container stays idle — developer starts the app via SSH.
      # Run: deno task dev (or: deno run --allow-net --allow-env src/main.ts)
      start: zsc noop --silent
```
